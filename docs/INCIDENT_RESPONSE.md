## 1. インシデント分類

### 1.1 重大度別分類

| 重大度 | 名称 | 定義 | SLA | 例 |
|--------|------|------|-----|-----|
| **Critical** | 緊急 | システム停止 or 大規模個人情報漏洩 or 児童虐待 | 30分 | DB ハッキング、CSAM 流出、サーバダウン |
| **High** | 重大 | 部分機能停止 or 限定個人情報漏洩 or 違法薬物販売 | 2時間 | 決済機能エラー、ユーザー100人のメール漏洩 |
| **Medium** | 中程度 | 軽微機能不具合 or 違法コンテンツ多発 | 8時間 | ログイン遅延、スパム 1000件 |
| **Low** | 軽微 | 非本質的不具合 or 単一違反コンテンツ | 24時間 | UI レイアウトエラー、ハラスメント投稿 1件 |

## 2. インシデント検出と初期対応

### 2.1 検出ソース

| ソース | 対応責任者 | 確認項目 |
|--------|----------|--------|
| **自動ログ監視** | DevOps | エラーログ、CPU/メモリ異常、DBジャム |
| **ユーザー報告** | カスタマーサポート | 通報・問い合わせ件数の多発 |
| **セキュリティツール** | セキュリティ | IDS/IPS アラート、WAF トリガー |
| **内部発見** | エンジニア | テスト環境での不具合 |
| **メディア・SNS** | マーケティング | 外部報道・話題化 |

### 2.2 初期トリアージ（5分以内）

インシデント検出時に以下を実行：

```
1. インシデント通知スレッド立ち上げ（Slack など）
   - タイトル: 「インシデント: [概要]」
   - 参加者: 関連する全メンバー（DevOps/セキュリティ/法務等）

2. 重大度の初期判定
   - システム影響? → Critical/High の可能性
   - ユーザー影響人数? → 1000人以上 → High
   - 個人情報漏洩? → High/Critical
   - 違法コンテンツ? → High（児童虐待は Critical）

3. インシデント番号を割り当て
   - 形式: INC-2026-001, INC-2026-002, ...
   - チケット（Jira等）を立ち上げ

4. リーダーを指名
   - 責任者: 重大度に応じて CEO / CTO / DevOps Lead
   - 責務: 対応指揮・報告書作成・ユーザー通知

5. タイマー開始
   - SLA タイマー開始
   - 30分ごとに進捗レポート
```

## 3. インシデント対応フロー

### 3.1 Critical インシデント（SLA: 30分）

**例: DB ハッキングで全ユーザーメール露出**

```
T+0min
├─ 初期トリアージ
├─ 緊急チーム招集（CEO/CTO/DevOps Lead/セキュリティ/法務）
└─ 被害範囲の初期見積

T+5min
├─ システム一時停止判断
│   ├─ YES → サービス全停止（外部通知）
│   └─ NO → 影響範囲を隔離・制限

T+10min
├─ 根本原因の調査開始
│   ├─ ハッキング? → フォレンジック準備
│   ├─ バグ? → コード確認
│   └─ 人的ミス? → プロセス検証

T+20min
├─ 対応実行決定
│   ├─ パスワードリセット強制（全ユーザー）
│   ├─ 不正アクセスがあったユーザーへの個別連絡
│   └─ 法執行機関への報告準備

T+30min
├─ ユーザーへの重要通知配信
│   └─ メール・アプリ内通知・SNS
└─ メディア対応準備

T+1-2時間
├─ 詳細原因分析
├─ 根本対策の実装
└─ システム復旧

T+24時間以内
├─ 全ユーザーへの正式報告書
├─ 個人情報漏洩の場合は個別通知
└─ 法的義務の履行確認
```

**通知内容テンプレート（例）:**
```
【重要】セキュリティインシデントのお知らせ

本日 2026-03-19 14:30 JST に弊社システムへの不正アクセスが発生いたしました。

影響範囲:
- 該当する可能性のあるユーザー: 全ユーザー（約 10万人）
- 露出した可能性のある情報: メールアドレス、ハンドルネーム

当社の対応:
1. 直ちにシステムを一時停止いたしました
2. 全ユーザーのパスワードをリセットいたしました
3. 法執行機関（警察庁）に報告いたしました
4. セキュリティレビューを実施中です

ユーザーの皆様へ:
- 次回ログイン時が新しいパスワード設定画面となります
- 心当たりのない操作があった場合は support@fiction-sns.example までご連絡ください
```

### 3.2 High インシデント（SLA: 2時間）

**例: ユーザー 500人からのスパム報告が短時間に多発**

```
T+0min
├─ 初期トリアージ
├─ チーム招集（DevOps Lead, モデレータ, セキュリティ）
└─ 被害背景のヒアリング

T+15min
├─ 根本原因の仮説
│   ├─ スパムボット発生?
│   ├─ 自動レート制限は機能している?
│   └─ アカウント乗っ取り多発?

T+45min
├─ 対応実行
│   ├─ 疑わしいアカウントの一時ロック
│   ├─ スパム投稿の一括削除
│   ├─ レート制限の強化（1h/10 投稿に制限）
│   └─ セキュリティチェック実施メール配信

T+120分以内
├─ 対応完了
├─ 原因報告書草稿
└─ ユーザーへの説明メール配信
```

### 3.3 Medium インシデント（SLA: 8時間）

ログイン遅延、スパム投稿多発など軽微なシステム不具合。

```
T+0h
├─ 検出・初期判定
└─ チーム内通知

T+1-4h
├─ 根本原因分析
├─ パッチ実装（または運用対応）
└─ テスト実行

T+4-8h
├─ 本番デプロイ
└─ ユーザーへのご報告（必要に応じて）
```

### 3.4 Low インシデント（SLA: 24時間）

軽微なバグ・単一違反コンテンツ。

```
標準的な開発プロセスに従う
├─ 修正コード実装
├─ レビュー → マージ
└─ 本番デプロイ（次回リリース時）
```

## 4. インシデントの通知・エスカレーション

### 4.1 通知チェーン

```
発見者
  ↓
DevOps Lead / セキュリティ（重大度判定）
  ↓
  ├─ Critical → CEO + CTO（直電話・Slack）
  │   └─ [並行] 法務・PR 通知
  │
  ├─ High → CTO（Slack + 電話要件）
  │   └─ [並行] DevOps Lead + モデレータ
  │
  ├─ Medium → DevOps Lead 判断
  │   └─ [並行] 関連チーム通知
  │
  └─ Low → Slack で静か通知
      └─ 標準プロセス適用
```

### 4.2 集約通知チャネル

| チャネル | 重大度 | 対象 |
|---------|--------|------|
| **Slack** #incidents | 全層 | 全インシデント |
| **メール** (全社) | Critical | CEO/全マネージャー |
| **紧急電話** | Critical | CEO/CTO |
| **外部通知** | High/Critical | ユーザー・メディア |

### 4.3 エスカレーション判定基準

以下の場合は 1 段階上に自動エスカレーション：

- **High → Critical 昇格** 条件:
  - SLA 内に対応できない見込み
  - 影響ユーザーが 10k 人以上に拡大
  - メディア報道が開始

- **Medium → High 昇格** 条件:
  - 根本原因が不明確で SLA 内終了不可
  - ユーザー影響が 1000 人以上に拡大

## 5. 記録・分析・改善

### 5.1 インシデント報告書

**対象**: Critical, High レベル

**報告書の項目:**
```
1. インシデント概要
   - ID: INC-2026-001
   - 発生日時, 検出日時, 対応開始時刻, 解決時刻
   - 重大度, 影響範囲（ユーザー数・機能等）

2. 根本原因
   - テクニカル分析
   - ビジネス/プロセス上の原因
   
3. 対応内容
   - 実施した対応（時系列）
   - 意思決定プロセス
   - ユーザー通知内容

4. 影響範囲と被害
   - ユーザー数
   - データ漏洩範囲（個人情報の場合）
   - システムダウン時間

5. 再発防止策
   - 実装予定の対策
   - 実装予定日
   - 責任者

6. レッスンズラーンド
   - うまくいったこと
   - 改善点
   - ネクストアクション
```

**報告書の完成期限:**
- Critical: インシデント終了から 3営業日以内
- High: 7営業日以内

### 5.2 インシデント振り返り会

**Critical/High インシデント 終了後 5営業日以内に開催**

参加者: CEO, CTO, DevOps Lead, セキュリティ, 関連エンジニア, 法務

議題:
- 根本原因の技術的分析
- 対応プロセスの評価
- 再発防止対策の承認
- ユーザー信頼回復策の検討

### 5.3 インシデント傾向分析

**月 1 回**: 全インシデントの集計・分析

分析項目:
- インシデントタイプ別集計（セキュリティ vs 機能 vs 運用）
- 検出から対応開始までの平均時間
- SLA 達成率
- 再発インシデントの有無

改善フィードバック:
- プロセス改善を提案・実装

## 6. 特殊なインシデント対応

### 6.1 個人情報漏洩インシデント

**定義:** ユーザーの個別特定情報（メール、実名など）が不正露出

**対応:**
1. **即座**:
   - 漏洩の止止（アクセス遮断）
   - 被害範囲の把握
   
2. **24時間以内**:
   - 該当ユーザーへの通知
   - 法執行機関への報告
   - クレジットカード会社との協力（決済情報の場合）
   
3. **30日以内**:
   - 詳細報告書提出（GDPRに準拠）
   - ユーザーへの信頼回復施策

### 6.2 セキュリティ脆弱性発見

**定義:** 製品内の セキュリティ脆弱性（未悪用）が発見

**対応:**
1. 直ちに脆弱性を隔離
2. 修正パッチ実装（48時間以内が目標）
3. 検証→ 本番デプロイ
4. ハッカーの倫理的報告対象の場合は謝金化

### 6.3 ユーザーアカウント乗っ取り（大規模）

**複数ユーザーが同時期にアカウント乗っ取りを報告**

```
T+0
├─ 被害ユーザーデータベース作成
├─ パスワードリセット強制
└─ 異常ログイン検跡から他の被害者を抽出

T+1-2h
├─ コード監査（認証ロジック確認）
├─ 外部ライブラリのセキュリティ確認
└─ API キー・秘密情報の確認

T+4h 以内
├─ 根本原因特定
├─ 被害ユーザー 100 人以上 → High 以上に昇格
└─ 全ユーザーへのパスワード変更通知（必要時）
```
